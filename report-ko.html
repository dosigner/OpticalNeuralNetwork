FD2NN / D2NN 광학 시스템 실험 ~18개 세션
회절 신경망(D2NN/FD2NN)을 활용한 광학 빔 정화, 대기 난류 보정, 메타렌즈 시스템 시뮬레이션에 대한 광범위한 작업. Claude Code를 사용하여 손실 함수 sweep 설계 및 실행, 물리 파이프라인(전파 방법, 빔 축소기) 디버깅, 파라미터 불일치 수정, 모델 학습, GPU 실험 모니터링을 수행. 물리적으로 비현실적인 파라미터와 파이프라인 버그로 인한 반복 수정이 주요 마찰점.
리포트 및 문서 생성 ~10개 세션
실험 결과를 문서화하는 종합 PDF 및 PPTX 리포트 생성. LaTeX 수식, 파라미터 표, 그림이 포함된 한/영 이중언어 PDF와 Obsidian 형식 마크다운 포함. Claude Code가 멀티파일 읽기, 그림 생성, LaTeX 렌더링, 누락된 이미지/폰트 문제/레이아웃 문제의 반복 수정을 담당.
데이터 시각화 및 분석 ~6개 세션
시각화 대시보드, 분석 그림, spacing 및 loss sweep 실험 결과의 물리 수준 해석 작성. Claude Code가 matplotlib 그림 생성, 메트릭 키 불일치 및 y축 스케일링 수정, 참조 논문과의 결과 비교, 발견 사항 요약 구조화 파일 생성.
Saliency Detection 모델 개발 ~5개 세션
ECSSD 데이터셋에서 D2NN 기반 saliency detection 모델 디버깅 및 개선. Claude Code가 blob 형태 출력 진단, 손실 함수 재설계(BCE+IoU+Structure+CenterPenalty), gamma_flip 및 crop 문제 수정, 배치 크기 실험 실행을 도와 체계적 디버깅으로 F_max를 ~0.55에서 ~0.59로 개선.
Claude Code 상태줄 및 환경 설정 ~5개 세션

Claude Code 사용 방식

당신은 FD2NN/D2NN 광학 물리 시뮬레이션 작업에 Claude Code를 풀스택 연구 보조원으로 활용하는 연구 중심 파워유저입니다. 세션 평균 ~5.7시간으로 길며 탐색적 성격이 강합니다 — sweep을 시작하고, GPU 작업을 모니터링하고, 물리 파이프라인을 디버깅한 뒤, 같은 세션에서 종합 리포트와 시각화를 요구합니다. 수정 사이클을 빠르게 반복하며, 물리 오류(잘못된 초점거리, 비현실적 NA 값, 잘못된 전파 파이프라인)를 직접 포착하고 Claude를 올바른 방향으로 이끕니다. 예를 들어, 전체 sweep이 잘못된 전파 방법(beam reducer 대신 zoom propagate)을 사용한 것을 발견하고, Claude가 놓친 pixel pitch 단위 오해를 교정했습니다.

상호작용 스타일은 높은 위임 + 높은 감독입니다. 리포트를 위한 5개 병렬 전문 에이전트 실행이나 9회 loss function sweep 등 야심찬 멀티스텝 작업을 시작하면서도, Claude가 이탈할 때를 포착할 만큼 충분히 관여합니다. 마찰 데이터가 명확한 이야기를 전합니다: 버그 코드(24건)와 잘못된 접근(19건)이 가장 큰 고충이지만, 세션을 포기하지 않고 반복적으로 수정합니다. 세련된 결과물(한/영 PDF, PPTX 프레젠테이션, 그림 포함 LaTeX 리포트)을 요청한 뒤 레이아웃 문제, 누락 이미지, 폰트 문제를 수정하는 QA 라운드를 진행하는 패턴도 눈에 띕니다. 40개 세션 중 8개만 완전 달성, 15개 부분 달성으로 Claude가 안정적으로 처리할 수 있는 경계에서 작업하고 있지만, 91건의 '만족 추정' 평가는 부분적 진전도 가치가 있음을 보여줍니다.

환경 커스터마이징(상태줄 조정, MCP 서버 정리, 플러그인 관리)에도 시간을 투자하여 워크플로우 도구를 최적화하려는 의지가 보입니다. 커밋이 0건인 것은 전통적 소프트웨어 개발이 아닌 실험/분석 맥락에서 작업하며, Claude Code를 코드 편집기보다 연산 노트북으로 활용하고 있음을 시사합니다.

핵심 패턴: 복잡한 물리 시뮬레이션과 리포트 작업을 위임하면서도 Claude의 기술적 실수를 실시간으로 교정하는 실전형 연구 PI로서 활동합니다.

인상적인 성과

지난 한 달간 물리 기반 광학 시뮬레이션 연구(FD2NN/D2NN)에 깊이 몰두하며, 40개 세션에 걸쳐 sweep 실행, 모델 디버깅, 완성도 높은 리포트를 생산했습니다.

반복적 물리 실험 파이프라인
손실 함수 설계부터 학습, 결과 분석까지 이어지는 광학 시뮬레이션 sweep의 정교한 워크플로우를 구축했습니다. 물리적으로 비현실적인 파라미터(잘못된 초점거리나 전파 방법 등)를 포착하고 수정을 이끄는 능력은 깊은 도메인 전문성과 Claude를 연구 파트너로 효과적으로 활용하는 역량을 보여줍니다.
다중 형식 리포트 생성
LaTeX 수식이 포함된 한/영 PDF, 37개 이상의 그림이 담긴 36페이지 리포트, PPTX 프레젠테이션, Obsidian 형식 마크다운 등 세련된 산출물을 꾸준히 생산합니다. 폰트 문제와 그림 렌더링을 반복 수정하여 학술 기준에 맞는 결과물이 나올 때까지 Claude Code를 완전한 출판 파이프라인으로 활용했습니다.

문제 발생 지점

물리적으로 잘못된 파라미터, 버그 있는 생성 코드, 의도와 맞지 않는 접근법으로 인해 세션이 자주 탈선하여 간단한 작업이 다중 수정 사이클로 변합니다.

잘못된 물리 및 파라미터 가정
Claude가 반복적으로 물리적으로 비현실적인 파라미터(초점거리, pixel pitch 단위, 정규화 방법)를 사용하여 사용자가 포착하고 수정해야 했으며, 상당한 반복 사이클이 낭비되었습니다. 참조 파라미터 파일 제공이나 CLAUDE.md에 검증된 값을 고정하면 이러한 반복적 오해를 방지할 수 있습니다.
  • 초기 광학 파라미터(f=1mm, NA=0.16, ROI crop 512)가 물리적으로 비현실적이어서 98% 에너지 손실을 야기, 사용자의 여러 차례 수정이 필요
  • Claude가 잘못된 초점거리를 사용하고 원시 빔 대신 D2NN 출력에서 기준 메트릭을 계산하여 매번 물리를 수동 교정해야 함
버그 있는 생성 코드
Claude가 생성한 코드에 버그(잘못된 메트릭 키, NaN 문제, 동시 파일 손상, PDF 렌더링 오류)가 자주 있어 런타임에서만 발견되며 반복적 디버그-수정 사이클을 강요합니다. Claude에게 작업 완료 선언 전 출력 검증이나 정합성 검사를 요청하면 줄일 수 있습니다.
  • 중복 데이터 생성 프로세스의 동시 쓰기로 인한 BadZipFile 오류로 sweep이 두 번 실패, 문제가 완전히 해결되지 않음
  • PPTX 생성에서 이미지 누락 및 가로세로비 왜곡, PDF 변환에서 그림 누락이 발생하여 매번 사용자가 문제를 지적해야 함
잘못된 접근법 또는 의도 오해
Claude가 근본적으로 잘못된 접근(잘못된 전파 파이프라인, 실행 대신 계획 수립, 시각화 요청 오해 등)을 자주 취하여 세션 중간에 방향 전환이 필요했습니다. 제약조건을 사전에 명시하면(예: 'zoom propagate 말고 beam reducer 사용') 이런 우회를 단축할 수 있습니다.
  • 전체 sweep이 근본적으로 잘못된 전파 파이프라인(beam reducer 대신 zoom propagate) 기반이어서 사용자가 발견할 때까지 모든 결과가 무효
  • 5개 병렬 에이전트로 리포트 작성을 요청했으나 Claude가 직접 실행 대신 리포트 구조 제안에 시간을 소비, 다른 세션에서는 종합 결과 대신 질문으로 끝남

물리 파라미터 오해가 5개 이상의 세션에서 주요 마찰을 일으켜 여러 차례 사용자 주도 수정과 sweep 낭비 발생.
리포트 생성이 7개 이상의 세션에서 주요 목표였으며 매번 동일한 문제(누락 그림, 폰트, 레이아웃 버그)로 여러 수정 라운드 필요.
여러 세션에서 동시 프로세스 충돌(BadZipFile 오류, GPU 경합)로 인한 sweep 실패로 상당한 컴퓨팅 시간 낭비.

Claude Code에 복사하면 자동으로 설정됩니다.

Hooks
특정 라이프사이클 이벤트에 자동 실행되는 셸 명령
추천 이유: 24건의 버그 코드 마찰과 19건의 잘못된 접근 문제가 있었습니다. 편집 전 Python 타입 검사를 실행하는 hook이나 편집 후 물리 파라미터 정합성 검사 hook을 설정하면 실패한 sweep으로 이어지기 전에 오류를 잡을 수 있습니다.
# .claude/settings.json에 추가: { "hooks": { "postToolUse": [ { "matcher": "Edit|Write", "command": "if echo \"$CLAUDE_FILE\" | grep -q '\\.py$'; then python -c \"import py_compile; py_compile.compile('$CLAUDE_FILE', doraise=True)\" 2>&1 || echo 'SYNTAX ERROR in edited file'; fi" } ] } }

40개 세션은 AI 보조 개발이 대화형 코딩에서 자율 실험 파이프라인으로 진화하는 정교한 연구 워크플로우를 보여줍니다. 파라미터 불일치, 버그 sweep, 미완성 실행에서 발생하는 마찰이 자율 에이전트가 다음 단계를 열 수 있는 정확한 지점을 보여줍니다.

자체 검증 기능이 있는 자율 실험 Sweep
가장 큰 마찰 원인 — BadZipFile 오류, 잘못된 물리 파라미터, 조용히 쓰레기를 생산하는 sweep — 은 각 단계에서 자체 출력을 검증하는 실험 sweep으로 제거할 수 있습니다. 자율 에이전트가 에너지 보존 확인, 전파 물리 검증, NaN/발산 감지, 실패한 실행 재시작을 사람의 개입 없이 수행하여 228시간의 모니터링을 실행-검토 워크플로우로 전환합니다.
시작 방법: Claude Code의 Agent 도구로 각 sweep 설정을 담당하는 서브에이전트를 생성하고, 부모 에이전트가 GPU 사용량 모니터링, 물리 제약 검증, 결과 종합을 담당합니다.
Claude Code에 붙여넣기:

Agent 도구로 각 sweep 설정에 병렬 서브에이전트 생성 3. 각 서브에이전트: 학습 시작, 30초마다 NaN/발산 모니터링, loss 폭발 시 LR 줄여 재시작, 체크포인트 저장 4. BadZipFile 또는 손상 오류 발생 시 손상 파일 삭제 후 데이터 재생성 후 재시도 5. 부모 에이전트: 모든 실행 추적, 마크다운 실시간 상태 대시보드 생성, 완료 시 최적 설정 강조 비교 표 생성 [경로]의 sweep 설정을 읽고 물리 검증부터 시작.
그림 포함 엔드투엔드 리포트 파이프라인
PDF/PPTX 리포트 반복에 엄청난 노력을 쏟고 있습니다 — 누락된 그림, 깨진 LaTeX, 한국어 폰트 문제, 잘못된 이미지 경로. 단일 프롬프트로 원시 실험 데이터에서 그림을 생성하고, 적절한 캡션과 상호 참조로 리포트를 조립하고, 누락 이미지와 렌더링 오류에 대한 QA 검사를 실행하며, EN/KO PDF를 한 번에 생산하는 자율 파이프라인을 트리거할 수 있습니다.
시작 방법: Claude의 Agent 도구를 Bash(LaTeX/matplotlib 렌더링)와 Read/Glob(실험 산출물 발견)과 체인하고, PDF를 열어 페이지 수와 이미지 존재를 확인하는 자체 QA 루프를 내장합니다.

세션에서 물리 버그를 뒤늦게 발견하는 사이클이 반복됩니다(잘못된 전파 파이프라인, 잘못된 Strehl 정규화, 상관 길이 불일치). 사후 발견 대신 병렬 전문 에이전트가 동시에 코드베이스를 논문 수식과 대조 감사하고, 파라미터를 출판된 값과 교차 검증하며, 진단 시각화를 실행하여 — GPU 시간을 결함 있는 실험에 낭비하기 전에 불일치를 포착합니다.
시작 방법: Claude의 Agent 도구로 각각 집중 감사 범위를 가진 3-4개의 전문 서브에이전트를 병렬 실행하고, 코디네이터가 발견 사항을 우선순위 액션 리스트로 종합합니다.

다음 실험 시작 전 D2NN 코드베이스 멀티에이전트 감사 실행: 에이전트 1 — 물리 검증기: 모든 전파/광학 코드를 읽고, 하드코딩된 모든 파라미터(pixel pitch, 파장, 초점거리, NA, 그리드 크기) 추출하여 검증: (a) 단위 일관성, (b) Nyquist 샘플링 충족, (c) 파이프라인 전체 에너지 보존. ./refs/의 참조 논문 값과 다른 파라미터 플래그. 에이전트 2 — 손실 함수 감사: 모든 손실 함수 구현을 읽고, gradient flow 검증(critical path에 detach() 없음), NaN 생성 edge case 확인(log(0), 0으로 나누기), 각 손실이 논문의 수학적 정의와 일치하는지 확인. 에이전트 3 — 데이터 파이프라인 점검: 학습 데이터 생성의 결정론적 여부 확인, 병렬 데이터 쓰기의 경쟁 조건 점검(이전에 BadZipFile 손상 발생), 전체 파이프라인의 입출력 텐서 형상 검증. 에이전트 4 — 결과 비교기: 최고 결과와 참조 논문의 Figure 4a/4b 메트릭을 로드, 차이 계산, 불일치의 가장 가능성 높은 원인 식별. 코디네이터: 모든 에이전트 보고서 수집, 영향도(GPU 시간 낭비 가능성) 기준 이슈 순위 지정, 우선순위 수정 목록 제공. 전체 감사를 ./audit/report.md에 저장.
"사용자의 물리 실험이 처음부터 잘못되었던 것이 밝혀짐 — Claude가 근본적으로 깨진 전파 파이프라인으로 sweep을 돌리고 있었다"
D2NN 빔 정화 실험 중, 사용자와 Claude는 초기 sweep 전체가 잘못된 전파 방법(beam reducer 대신 zoom propagate)을 기반으로 했음을 발견했습니다. 그 모든 GPU 시간, 그 모든 분석 — 잘못된 물리 위에 세워진 것이었습니다. 올바른 beam reducer 구현으로 처음부터 다시 시작해야 했습니다.
